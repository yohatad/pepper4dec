## `cpp` branch (WIP C++ port)

No regression safety net for ONNX inference glue, Hungarian face/person matching, ByteTrack integration, or the 1702-line BT node library in `behavior_controller_implementation.cpp`. Start with `gesture_execution/src/pepper_kinematics_utilities.cpp` — pure math, cheapest to unit-test.

2. ~~**Duplicated, error-swallowing config loading.**~~ — **Fixed.** `face_detection` and `person_detection` now declare real ROS2 parameters (via a new shared `dec_common::declareAndGetParameter` helper) in `on_configure()` instead of manually parsing YAML with try/catch; `ros2 param get/set` works on both now, and config files were renamed to snake_case `ros__parameters` blocks to match `animate_behavior`/`overt_attention`. Also fixed a latent bug found along the way: `face_detection_configuration.yaml`'s `objectDetectionTimeout` key didn't match what the old parser read (`personDetectionTimeout`), so it silently never took effect.

3. ~~**`std::cout`/`std::endl` instead of RCLCPP logging**~~ — **Fixed** for the sites this item named. `face_detection`/`person_detection`'s config-loader `std::cout` calls were removed as part of item 2; the two remaining `getClassIndices()` warnings in `person_detection_implementation.cpp` now take an `rclcpp::Logger` and use `RCLCPP_WARN`; and the `*_application.cpp` startup banners in `face_detection`, `person_detection`, and `gesture_execution` now use `RCLCPP_INFO` (verified live — timestamps/logger name show up correctly). Note: `pepper_odom_anchor`/`animate_behavior` have no banner at all and `behavior_controller`'s banner already used `RCLCPP_INFO`, so the original "six packages" scope was smaller than stated.

4. **Six near-identical `*_application.cpp` files** (person_detection, face_detection, gesture_execution, pepper_odom_anchor, animate_behavior, behavior_controller) hand-roll `rclcpp::init` → construct → spin → `shutdown` boilerplate, with ad hoc single- vs multi-threaded executor choice. Worth a shared helper.

5. ~~**Unit-mixed magic numbers in kinematics**~~ — **Fixed.** `gesture_execution/src/pepper_kinematics_utilities.cpp:35,71` used meters (`l_6 = 0.13`) while `:150,191` used millimeters (`l_4 = 150.0`, `d_5 = 150.0`) hardcoded inline with no named/unit-labeled constants. Values are unchanged (verified against the `main`-branch Python reference — this is a faithful port, not a porting bug) but are now named `constexpr` constants with units in the name/comments (e.g. `kElbowOffsetXM`, explicitly called out as meters), so the mm/m mix can't be silently "corrected" away.

6. **No intra-process communication.** All nodes are standalone executables launched as separate processes via plain `launch_ros.actions.Node(...)` — no `NodeOptions().use_intra_process_comms(true)`, `rclcpp_components`, or `ComposableNodeContainer`. Worth adding for the image-heavy perception pipeline (face_detection, person_detection, overt_attention) where `sensor_msgs::msg::Image` frames are currently published by dereferenced value (`pub_->publish(*msg)`) rather than moved `unique_ptr`. Skip for control/behavior nodes (gesture_execution, behavior_controller) — low payload/rate, not worth the debuggability cost of composing them into one process.

   **Investigated and deferred (2026-07-08):** every Image-consuming node here — `face_detection`, `person_detection`, `overt_attention_saliency`, `overt_attention_visualization` — subscribes directly and independently to the same external camera-driver topic (`topics_config_.image.pepper`/`.realsense`); none of them receive an `Image` from another in-repo node (`face_detection`'s only cross-node subscription is `person_detection`'s structured `PersonDetection` message, not an image). So the real image-copy fan-out is driver → 4 independent subscribers, and composing only *our* nodes into one container doesn't touch that — the driver stays out-of-process either way. Fixing it for real would require the camera driver itself in the container, which isn't feasible for `naoqi_driver` (the default `camera: "pepper"` path) and is unverified for `realsense2_camera`. Not worth the complexity (shared-lib restructuring, container launch wiring, lifecycle-in-container debugging) for no measurable win on the default hardware path. Revisit only if `realsense2_camera`'s composability is confirmed useful and the RealSense path becomes primary.

1. **Near-total duplication between face_detection and person_detection (highest impact).** `face_detection/face_detection/face_detection_implementation.py` (1333 lines) and `person_detection/person_detection/person_detection_implementation.py` (996 lines) share an almost identical lifecycle-node skeleton — `load_configuration`, `on_configure/activate/deactivate/cleanup/shutdown`, `update_latest_frame`, `visualization_callback`, `get_topic_names`, `extract_topic`, `wait_for_topics`, `subscribe_topics`, `synchronized_callback`, `rgb_only_callback`, `process_depth_image`, `start_timeout_monitor`/`check_timeout`, `check_camera_resolution`, `make_depth_vis`, `get_depth_in_region`, `generate_dark_color`, `cleanup` — copy-pasted with only cosmetic renaming. 4. **`print()` instead of the rclpy logger at config-load time**, e.g. `face_detection_implementation.py:84-85`, `person_detection_implementation.py:97-98` (`load_configuration`, runs before the node/logger exists), and `person_detection_application.py:90-91,101-102` in `main()`'s try/except.
